## Rust 宏：从预处理器到元编程

> **你将学到什么：** Rust 宏如何工作，何时使用它们而不是函数或泛型，以及它们如何替代 C/C++ 预处理器。学完本章你可以编写自己的 `macro_rules!` 宏并理解 `#[derive(Debug)]` 底层做了什么。

宏是你在 Rust 中首先遇到的东西之一（第一行就是 `println!("hello")`）但大多数课程最后才解释的东西。本章解决这个问题。

### 为什么宏存在

函数和泛型处理 Rust 中的大多数代码重用。宏填补类型系统无法触及的空白：

| 需求 | 函数/泛型？ | 宏？ | 为什么 |
|------|-------------------|--------|-----|
| 计算值 | ✅ `fn max<T: Ord>(a: T, b: T) -> T` | —— | 类型系统处理它 |
| 接受可变参数 | ❌ Rust 没有变参函数 | ✅ `println!("{} {}", a, b)` | 宏接受任意数量的 token |
| 生成重复的 `impl` 块 | ❌ 泛型单独无法做到 | ✅ `macro_rules!` | 宏在编译时生成代码 |
| 编译时运行代码 | ❌ `const fn` 有限 | ✅ 过程宏 | 完整 Rust 代码在编译时运行 |
| 条件包含代码 | ❌ | ✅ `#[cfg(...)]` | 属性宏控制编译 |

如果你来自 C/C++，把宏想象成 *预处理器唯一正确的替代品* —— 只是它们在语法树上操作而不是原始文本，所以它们是卫生的（无意外名称冲突）且类型感知。

> **面向 C 开发者：** Rust 宏完全替代 `#define`。没有文本预处理器。参见 [ch18](ch18-cpp-rust-semantic-deep-dives.md) 了解完整的预处理器 → Rust 映射。
